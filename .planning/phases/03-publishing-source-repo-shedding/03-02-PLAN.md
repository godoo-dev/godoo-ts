Configure npm OIDC trusted publishers for the three @godoo-dev/* stub packages so that when the release workflow runs on push to main it can publish without a long-lived token.

Output: Three trusted publisher configurations on npmjs.com; release/stub-bootstrap branch cleaned up.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| npmjs.com trusted publisher config | Binds OIDC subject to specific repo+workflow; misconfiguration allows unauthorized publish |
| GitHub OIDC provider → npm registry | OIDC token scoped to specific workflow run; short-lived |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-05 | Elevation of Privilege | Trusted publisher misconfiguration (too-broad subject claim) | mitigate | Bind trusted publisher to specific workflow file (release.yml) and specific repo (godoo-dev/godoo-ts); do not use wildcard subject |
| T-03-06 | Elevation of Privilege | Missing "allow npm publish" action post-May-20-2026 | mitigate | Pitfall 4 explicitly documented: must check the "npm publish" checkbox when configuring; CLI equivalent: --allow-publish flag |
| T-03-07 | Spoofing | Forked workflow triggering OIDC publish | mitigate | Trusted publisher is scoped to repo:godoo-dev/godoo-ts only; forks have a different OIDC subject claim (repo:<fork-owner>/<fork-name>) and cannot satisfy the trust check |
| T-03-08 | Tampering | Stale release/stub-bootstrap branch creating confusion | mitigate | Task 2 deletes the branch immediately after configuration; stubs never land on develop or main |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in workspace — no install audit required for this plan |
</threat_model>
